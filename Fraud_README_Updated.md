# 🚀 Deep Hybrid Fraud Detection System

A hybrid fraud detection system for **real-time credit card anomaly detection** using **Deep Learning + Machine Learning + MLOps**.
The project is designed to detect fraudulent transactions — including **previously unseen patterns** — through an **unsupervised anomaly detection pipeline** that does not rely on labeled fraud samples for training.

---

## 📌 Project Overview

Fraud detection is a highly imbalanced classification problem where fraudulent transactions represent a tiny fraction of total activity, yet missing them can be extremely costly. Traditional supervised approaches often depend on historical labeled fraud cases, which makes them less effective against **new or evolving fraud strategies**.

This project addresses that challenge through a **hybrid unsupervised architecture** that combines:

* **Autoencoder** to learn normal transaction behavior and measure reconstruction error
* **Isolation Forest** to detect anomalies in latent feature space
* **Decision fusion logic** to maximize fraud coverage in real-time inference

The result is an end-to-end fraud detection pipeline that emphasizes **high recall, low-latency scoring, and production-oriented model management**.

---

## 🛡️ Anti-Data Leakage Architecture (The Data Firewall)

To transition from a "notebook research" script to an enterprise production pipeline, this system implements a strict **Data Firewall** across the entire pipeline to completely eliminate **Data Leakage**:

* **Strict Stratified Split:** Holdout validation and test sets are partitioned before any mathematical transformations occur to prevent future-data contamination.
* **Isolated Feature Transformations:** The Log Transformation (`np.log1p`) used to fix feature skewness on the `Amount` feature is applied strictly within split boundaries.
* **Template-Based Scaling:** The `StandardScaler` computes its statistical parameters ($μ, σ$) strictly from the training set, using them as an immutable template to `transform` validation and test sets independently.
* **Validation Leakage Defeated:** Zero cross-contamination during Deep Learning warm-ups by passing manual explicit validation pairs instead of automated random split ratios.

## 🧠 System Architecture

The system follows a **two-stage unsupervised pipeline**:

### 1) Autoencoder

A deep autoencoder is trained to reconstruct normal transaction behavior.
Transactions with **high reconstruction error** are treated as suspicious.

### 2) Isolation Forest

The compressed latent representation generated by the autoencoder is passed to an **Isolation Forest**, which identifies geometrically isolated anomalies in feature space.

### 3) Final Fraud Decision

The final prediction is based on an **OR decision rule**:

* If **Autoencoder** flags the transaction → **Fraud**
* If **Isolation Forest** flags the transaction → **Fraud**
* If **both models pass** → **Legitimate**

This design increases fraud coverage by combining:

* **Behavioral anomaly detection** (Autoencoder)
* **Geometric anomaly detection** (Isolation Forest)

## 📊 Model Performance

Evaluation was performed on the credit card fraud dataset containing **284,807 transactions**.

| Metric                  |      Value |
| ----------------------- | ---------: |
| **Fraud Recall**        | **83.33%** |
| **Fraud Precision**     |  **2.03%** |
| **Accuracy**            | **93.59%** |
| **False Negative Rate** | **16.67%** |

### Interpretation

* The system detects **5 out of every 6 fraudulent transactions**
* Precision is intentionally low due to **extreme class imbalance**, which is common in fraud detection
* In this domain, **Recall is the primary business metric**, because missing fraud is often far more expensive than reviewing false positives

---

## 📈 Why Recall Matters More Than Precision in Fraud Detection

Fraud detection is not a balanced classification problem.

* **False Negative (missed fraud)** → can result in financial loss, chargebacks, and reputational risk
* **False Positive (legitimate transaction flagged)** → usually results in manual review or minor customer friction

Because of that, the project is optimized around **high recall**, even if that comes with more alerts for downstream review.

## 🔍 Key Data Insights

During analysis, several patterns motivated the hybrid design:

* **Extreme class imbalance**: fraud cases represent **less than 0.2%** of all transactions
* **Fraud can mimic normal behavior**: simple rule-based detection on transaction amount is not sufficient
* **Behavioral irregularities matter**: fraud often differs through subtle multi-feature interactions rather than one obvious variable
* **Temporal irregularities exist**: suspicious activity may cluster during off-hours
* **Fraud forms detectable anomaly structure** in feature space, which supports latent-space anomaly detection

---

## 🧪 Why a Hybrid Approach?

A single anomaly detector can miss certain fraud patterns.

### Autoencoder strengths

* Learns the structure of normal behavior
* Captures non-linear feature relationships
* Detects suspicious deviations via reconstruction error

### Isolation Forest strengths

* Detects isolated anomalies geometrically
* Works efficiently on compressed latent features
* Adds a second detection perspective beyond reconstruction error

### Why combine them?

Using both models improves robustness by covering **different anomaly mechanisms**:

* the **Autoencoder** focuses on reconstruction behavior
* the **Isolation Forest** focuses on feature-space isolation

This makes the system more effective than relying on a single unsupervised detector alone.
